## Above-the-fold order (per the demand-side scan)

The screen reads the **first case study's header block** and rarely past 3–4 case studies, never to the bottom. So:

1. **Hero** — name, "Group PM, Enterprise", level-legible in one line; the `stat-row` directly beneath it so the deltas land inside the F-pattern
2. **Experience** (default-open pane) — newest first; each card header front-loads Problem / My Role / Key Outcomes; the decision narrative (the killed tour feature, the failed checkout variants) lives inside the expanded card — both-ends rule
3. Projects → Skills, education last

Bin signals avoided: zero vague "improved engagement" claims (every delta has a period attached inside the card), no feature lists without business outcome, no process prose detached from artifacts — the PRD teardowns are the artifacts.

## Extrapolation notes — what generalizes vs what is Priya-specific

1. **Generalizes:** relative deltas for confidential metrics — the pack converts absolutes to percentages for every PM, not just NDA-shy ones. Hope asks before assuming a number is public.
2. **Generalizes:** the 3–4 case-study ceiling and Problem / My Role / Key Outcomes header order — that's the demand-side read, independent of persona.
3. **Priya-specific:** the `stat-row` shipping at all — it's the pack's *optional* app and only earns its place because she has four clean deltas. A first-time PM with one launch skips it (R14 dosage).
4. **Priya-specific:** redacted PRDs as public projects. The redacted-with-permission norm generalizes; *having* cleared permissions does not — Hope verifies the user actually has them before composing this card.
5. **Generalizes (seniority modulation):** at exec level the spec inverts this composition — leadership narrative leads and project artifacts dilute. Priya is one rung below that line; the composition would change with her next promotion.
