## Above-the-fold order (per the demand-side scan)

The scan checks **project titles for real-world vs toy problems, then opens exactly ONE README** — no README, no further look. So:

1. **Hero** — name, "Senior data scientist · forecasting", and three metric stats (WAPE −23% · AUC 0.81→0.86 · MAPE 4.2% vs 6.8%)
2. **Projects** (default-open pane) — titles name real systems (311, ERCOT, CMS), never datasets-as-toys; the card body mirrors a strong README: problem / data / approach / results + limitations, dirty-data step included
3. Skills → Experience → Education; Talks & notebooks below the fold

Bin signals avoided by construction: no Titanic/MNIST/iris anywhere, not Kaggle-led, no card starts at the modeling step, three distinct techniques (geospatial EDA, gradient-boosted forecasting, hierarchical time series) — no one-hammer, and no case study without code behind it.

## Extrapolation notes — what generalizes vs what is Diego-specific

1. **Generalizes:** the proprietary-data gate's move — employer data never ships; the *technique* is rebuilt on public/synthetic data and labeled as a rebuild. Every data persona gets this, hardest in healthcare/finance.
2. **Generalizes:** the README grammar (problem / data / approach / results + limitations) as the card body — the demand side opens one README and judges the lot by it.
3. **Diego-specific:** the limitations-forward, "what didn't work" register — it reads senior because he *is* senior. The spec's junior modulation flips to fundamentals-end-to-end on one real problem instead.
4. **Diego-specific:** the "Talks & notebooks" list shipping — it's the pack's only optional app and needs real talks to exist (R15). No talks, no app.
5. **Generalizes:** one chart image per project card, never more — the dosage discipline (R14) holds whatever the persona's chart supply looks like.
